## 一、三层结构总览

```
Project（项目 = 一个故事）
├── writing_rules: 写作要求（约束 AI 的规则）
├── world_settings: 世界设定
│   ├── settings[]: 组织、场景、地理等设定
│   └── world_rules[]: 世界法则（魔法体系/时代背景/物理规则）
│
├── Events[]: 事件图（节点 + 边）
│   └── 每个事件 = 一个有向图节点
│       事件可通过 sub_events[] 包含子事件，形成树形层次
│
└── Entities[]: 实体库（演员）
    └── 每个实体 = 一个人物/物品/概念
```

## 二、事件（Event）

事件是"发生了什么"。通过入边（前因）和出边（后果）连接成因果网络。事件也可以通过 `sub_events[]` 包含子事件，形成层次结构。

### 2.1 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| **id** | string | ✅ | 唯一标识，如 `ev_001` |
| **name** | string | ✅ | 名称，展示给用户，如"受封岑州" |
| **introduction** | string | ✅ | 介绍：这个事件的意义、核心内容概括 |
| **type** | enum | ✅ | `event`（普通）/ `container`（容器，自身为标题）/ `mixed`（既有自身内容又有子事件） |
| **time** | any | 可选 | 时序标识：事件在故事时间轴上的位置（格式由项目自定义） |
| **setting_ref** | string | 可选 | 设定归属：关联于哪个场景/组织/设定域。约束 AI 不要跨域编造 |
| **in_edges[]** | array | 可选 | 入边（前因）：什么导致了这件事——指向事件 ID |
| **out_edges[]** | array | 可选 | 出边（后果）：这件事导致了什么——指向事件 ID |
| **sub_events[]** | array | 可选 | 子事件列表：这个事件包含的子事件 ID（有序），形成树形包含关系 |
| **participants[]** | array | 可选 | 参与者列表（见下方详解） |
| **process** | string | ✅ | 过程：尽可能详尽的事件描述（AI 生成时的核心输入） |
| **outcome** | string | ✅ | 结果：事件造成的影响——哪些实体变了、世界状态怎么变了 |
| **locked** | bool | 可选 | 是否锁定（锁定后 AI 不可修改），默认 `false` |

### 2.2 三种事件类型

| type | 含义 | 渲染方式 | 示例 |
|------|------|----------|------|
| **event** | 普通事件，没有子事件 | 渲染为内容块（###），无递归 | "遇刺"、"朝堂对峙" |
| **container** | 纯容器事件，自身没有内容，只作为子事件的标题/分组 | 渲染为标题（#），递归处理 sub_events | "第一卷·风起" |
| **mixed** | 既有自己的内容，又包含子事件 | 渲染为标题（#）+ 自身内容，递归处理 sub_events | "酒馆冲突"（父事件有概括，子事件有细节） |

> container 类型就是"故事段"的替代。不需要单独的 StorySegment 类型。

### 2.3 事件 JSON 示例

```json
{
  "id": "ev_tavern",
  "name": "酒馆冲突",
  "introduction": "在酒馆中与宿敌陈豹对峙并冲突的事件",
  "type": "mixed",
  "time": [24, "白华四年", "冬"],
  "setting_ref": "set_cen_state",
  "sub_events": ["ev_enter", "ev_provoke", "ev_confront", "ev_fight"],
  "in_edges": [{ "target": "ev_before_tavern", "type": "cause" }],
  "out_edges": [{ "target": "ev_after_tavern", "type": "result" }],
  "participants": [
    {
      "entity_id": "ch_lz",
      "state": { "mood": "谨慎而忐忑", "physical": "轻伤未愈" },
      "impact": "意识到暴力在当地政治中的必要性"
    }
  ],
  "process": "李葬推门进入酒馆，嘈杂声顿止…",
  "outcome": "与陈豹结下死仇，但赢得了当地势力的初步尊重",
  "locked": false
}
```

### 2.4 in_edges / out_edges 设计意图

```
[事件A] ──出边──→ [事件B] ──出边──→ [事件C]
   ↑                  ↑
   └────── 入边 ──────┘
```

- **入边**回答"为什么发生"（前因），**出边**回答"然后呢"（后果）
- 多个出边 = 分支叙事（多线叙事、平行可能性）
- 多个入边指向同一节点 = 叙事线汇聚
- 出边数组的第一个 ID 通常是主链顺序，后续 ID 是分支

### 2.5 sub_events：包含关系

除了因果链，事件还可以通过 `sub_events[]` 包含子事件。这是**整体/部分**关系，不是因果关系。

```json
// 父事件
{
  "id": "ev_tavern",
  "name": "酒馆冲突",
  "type": "mixed",
  "sub_events": ["ev_enter", "ev_provoke", "ev_confront", "ev_fight"],
  "in_edges": [{ "target": "ev_before_tavern", "type": "cause" }],
  "out_edges": [{ "target": "ev_after_tavern", "type": "result" }],
  "process": "这是酒馆冲突的概括性描述…"
}

// 子事件之间仍有因果
{
  "id": "ev_enter",
  "name": "进门",
  "sub_events": [],
  "in_edges": [],
  "out_edges": [{ "target": "ev_provoke", "type": "result" }]
}
{
  "id": "ev_fight",
  "name": "动手",
  "sub_events": [],
  "in_edges": [{ "target": "ev_confront", "type": "cause" }],
  "out_edges": [{ "target": "ev_after_tavern", "type": "result" }]
}
```

### 2.6 参与者（Entity Event Participation）

参与者是**事件和实体之间的中间表**，记录：**谁 + 什么状态 + 受到了什么影响**。

```json
{
  "entity_id": "ch_lz",
  "state": {
    "mood": "谨慎而忐忑",
    "physical": "轻伤未愈"
  },
  "impact": "从中枢边缘人被推向现实治理"
}
```

| 子字段 | 类型 | 说明 |
|-------|------|------|
| **entity_id** | string | 关联的实体 ID |
| **state** | object | 参与时的状态（心情、身体状况等）—— 记录"入场快照" |
| **impact** | string | 事件对该实体的影响 —— 记录"出场变化" |

> 事件 `outcome` 描写事件本身的结果，参与者的 `impact` 是描写人物的变化，二者重心不同。

所有参与者的 `state` 按时间排列，就是这个实体的**角色弧光**。

## 三、实体（Entity）

所有可能在剧情中出现的对象：人物、物品、组织、概念。

### 3.1 字段定义

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| **id** | string | ✅ | 唯一标识，如 `ch_lz` |
| **name** | string | ✅ | 名称，展示给用户 |
| **type** | string | ✅ | 类型标注：人物、物品、组织等 |
| **introduction[]** | array | ✅ | 介绍：性格、身体状况、能力、背景故事等，以条记录方便局部修改 |
| **relationships[]** | array | 可选 | 关系列表（见下方详解） |
| **rule_refs[]** | array | 可选 | 关联规则设定：和某些世界观概念/规则的联系 |

### 3.2 关系（Entity Relationship）

关系是**实体和实体之间的中间表**，记录：**谁 + 和谁 + 什么关系**。

```json
{
  "target_id": "ch_wind_sprite",
  "relation_type": "同伴",
  "description": "风精灵是他的伙伴，会在危险时保护他"
}
```

| 子字段 | 类型 | 说明 |
|-------|------|------|
| **target_id** | string | 目标实体 ID |
| **relation_type** | string | 关系类型（同伴/敌对/家人/师徒等） |
| **description** | string | 关系描述 |

- 和参与者的设计思路一致：把多对多关系展开，在边上附加属性
- 关系可以双向，也可以单向（如暗恋）

### 3.3 实体 JSON 示例

```json
{
  "id": "ch_allan",
  "name": "Allan",
  "type": "人物",
  "introduction": [
    "穿越者，经历过濒死，有PTSD。",
    "左手骨折已结痂，裤管空但能感觉到腿（解离）。",
    "有风精灵同伴。"
  ],
  "relationships": [
    {
      "target_id": "ch_wind_sprite",
      "relation_type": "同伴",
      "description": "风精灵是他的伙伴，会在危险时保护他"
    }
  ],
  "rule_refs": []
}
```

## 四、三种关系详解

本数据模型中有三种正交的关系：

| 关系类型 | 含义 | 表达方式 | 作用域 | 回答的问题 |
|---------|------|---------|--------|-----------|
| **因果** | 事件 A 导致了事件 B | `in_edges[]` / `out_edges[]` | 事件之间 | "为什么发生" |
| **包含** | 事件 A 包含事件 B（整体/部分） | `sub_events[]` | 事件树内部 | "属于哪个部分" |
| **参与** | 实体参与了事件（含状态、影响） | `participants[]` | 事件 ⇄ 实体 | "谁在场、什么状态" |

### 4.1 为什么区分这三种关系

| 如果用一种关系替代 | 出现的问题 |
|-------------------|-----------|
| 用 `out_edges` 表达包含 → 子事件 | 混淆了"A 导致了 B"和"B 是 A 的一部分"——遍历时无法区分是执行子事件还是跳转到下一个因果节点 |
| 用独立 Segment 类型表达包含 | 需要维护两套树（段树 + 事件图），且段不能有 participants/process/outcome，功能受限 |
| 不记录参与者状态 | 无法追踪角色弧光（角色的状态变化历史） |

**结论**：三种关系完全正交，各司其职，共同构成故事的全息结构。

### 4.2 关系正交示意图

```
    包含树                   因果链                    角色弧光
    ───────                  ───────                   ───────
  第一卷·风起
  ├── 第一章·入京            ev_出发 → ev_遇刺         出发时 {mood:"期待"}
  │   ├── ev_出发                    ↓                 遇刺时 {mood:"恐惧", physical:"受伤"}
  │   ├── ev_遇刺            → ev_朝堂                朝堂时 {mood:"谨慎"}
  │   └── ev_朝堂                  ↓
  └── 第二章·立威            ev_新政 → ev_受阻
      ├── ev_新政
      └── ev_受阻
```

- **竖着看**：包含关系（sub_events）
- **横着看**：因果关系（in_edges/out_edges）
- **沿着实体看**：状态变化轨迹（participants[].state）

## 五、叙事顺序与时间轴

故事中存在两种顺序：

| 维度 | 含义 | 记录方式 |
|------|------|----------|
| **故事时间轴** | 事件真实发生的先后 | `time` + 入边/出边 共同确定 |
| **叙事链** | 读者看到的顺序（可重排，支持倒序插叙等非线性叙事） | 在 Project 层用 `spine[]` 数组记录 |

### 5.1 spine 与因果链的关系

**生成 MD 大纲/正文时**，按 `spine` 数组顺序遍历。

**评估事件因果合理性时**，按 `in_edges` / `out_edges` 验证。

如果 `spine` 顺序违反因果逻辑，标记警告但不阻止——这是非线性叙事（倒叙、闪回、插叙）的实现基础。

### 5.2 time 字段设计

`time` 字段**完全开放**，由用户自定义。通常是一个数组：

```json
// project.json 中的时间系统定义
"time_system": ["age", "era", "season"]

// 事件中的 time 字段
"time": [21, "白华元年", "春"]
```

> 时间系统的格式完全由 `project.json` 中的 `time_system` 数组决定。用户可以用年龄/纪元/季节/任意维度。

## 六、从树到 MD 大纲的生成

事件通过 sub_events 形成树，但最终的 MD 大纲是线性的。生成规则：

> **递归遍历事件树：遇到有子事件的事件，渲染它为标题，然后递归渲染其子事件。遍历的深度（depth）就是标题的 `#` 个数。**

### 6.1 渲染规则

| type | 渲染方式 |
|------|----------|
| container | 渲染为标题（`# x depth`），递归处理 sub_events |
| mixed | 渲染为标题 + 自身内容，递归处理 sub_events |
| event | 渲染为内容块（最小 `###` 开头），不递归 |

### 6.2 示例

```
事件树：
ev_vol_1（第一卷·风起，type=container，depth=0）
├── ev_ch_1（第一章·入京，type=container，depth=1）
│   ├── ev_001（出发，type=event，depth=2）
│   └── ev_002（遇刺，type=event，depth=2）
└── ev_ch_2（第二章·立威，type=container，depth=1）
    └── ev_003（朝堂对峙，type=event，depth=2）
ev_004（后续事件，type=event，depth=0）
```

深度优先遍历后：

```markdown
# 第一卷：风起                  ← container，depth=0 → `#`

## 第一章：入京                  ← container，depth=1 → `##`

### 出发                        ← event，depth=2 → `###`
李葬收拾行囊…

### 遇刺
路上遭遇伏击…

## 第二章：立威

### 朝堂对峙
满朝文武…

### 后续事件                    ← event，depth=0 → `###`
接下来发生的事…
```

### 6.3 实现逻辑（伪代码）

```
function renderOutline(events, spine):
    for eventId in spine:
        event = load(eventId)
        renderEvent(event, depth=0)

function renderEvent(event, depth):
    if event.type == 'container':
        printHeader(event.name, depth)
        for subId in event.sub_events:
            renderEvent(load(subId), depth+1)

    elif event.type == 'mixed':
        printHeader(event.name, depth)
        printContent(event.process, depth)
        for subId in event.sub_events:
            renderEvent(load(subId), depth+1)

    else:  // type == 'event'
        printHeader(event.name, depth+2)
        printContent(event.process, depth)
        printParticipants(event.participants)
```

## 七、数据模型全貌图

```
Project
├── writing_rules: 写作要求/约束
├── world_settings:
│   ├── settings[]: 组织、场景、地理设定
│   └── world_rules[]: 世界法则
│
├── time_system: 时间系统定义
├── spine[]: 叙事链顺序（事件 ID 数组，控制读者看到顺序）
│
├── Events[]: 事件列表
│   └── Event
│       ├── id, name, introduction, time, type
│       ├── setting_ref: 设定归属（→ world_settings）
│       ├── in_edges[]: 入边（→ Event.id，因果关系）
│       ├── out_edges[]: 出边（→ Event.id，因果关系）
│       ├── sub_events[]: 子事件列表（→ Event.id，包含关系）
│       ├── participants[]:
│       │   ├── entity_id (→ Entity.id)
│       │   ├── state: {mood, physical, ...}
│       │   └── impact: 事件对该实体的影响
│       ├── process: 详尽过程描述
│       ├── outcome: 结果/影响描述
│       └── locked: 是否锁定
│
└── Entities[]: 实体列表
    └── Entity
        ├── id, name, type, introduction[]
        ├── relationships[]:
        │   ├── target_id (→ Entity.id)
        │   ├── relation_type, description
        ├── rule_refs[]: 关联规则设定
        └── events[]: 参与事件索引（→ Event.id）
```

## 八、文件结构规范

```
project_{id}/
├── project.json              # 项目元信息 + spine + 世界观规则 + 时间系统
├── nodes/                    # 事件节点目录
│   ├── ev_001.json
│   ├── ev_002.json
│   ├── ev_vol_1.json         # 容器事件（相当于"卷"）
│   ├── ev_ch_1.json          # 容器事件（相当于"章"）
│   └── ...
├── chars/                    # 实体（角色）节点目录
│   ├── ch_lz.json
│   └── ...
└── assets/                   # 附件（可选）
    ├── original_text.md
    └── generated/
        ├── chapter_01.md
        └── ...
```

> 容器事件和普通事件都是 Event，共享同一个 `nodes/` 目录。区别只在于 `type` 和 `sub_events` 字段。

## 九、数据库设计

### 9.1 核心表

#### projects 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(64) PK | 项目唯一标识 |
| title | VARCHAR(255) | 项目名称 |
| author | VARCHAR(128) | 作者 |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |
| spine | JSON | 叙事链顺序 ID 数组 |
| world_rules | JSON | 世界观规则列表 |
| time_system | JSON | 时间系统定义 |
| settings | JSON | 项目设置（模型、缓存等） |
| writing_rules | JSON | 写作约束规则 |

#### events 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(64) PK | 事件 ID |
| project_id | VARCHAR(64) FK | 所属项目 |
| name | VARCHAR(255) | 事件名称 |
| introduction | TEXT | 介绍/摘要 |
| type | ENUM('event','container','mixed') | 事件类型 |
| time | JSON | 时间标记 |
| setting_ref | VARCHAR(64) | 关联设定 |
| in_edges | JSON | 入边列表 [{target, type}] |
| out_edges | JSON | 出边列表 [{target, type}] |
| sub_events | JSON | 子事件 ID 数组 |
| participants | JSON | 参与者列表 |
| process | TEXT | 事件过程描述 |
| outcome | TEXT | 事件结果 |
| locked | BOOLEAN | 是否锁定 |
| sort_order | INTEGER | spine 中的位置（冗余，用于排序） |

#### entities 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(64) PK | 实体 ID |
| project_id | VARCHAR(64) FK | 所属项目 |
| name | VARCHAR(255) | 实体名称 |
| type | VARCHAR(64) | 类型（人物/物品/组织等） |
| introduction | JSON | 介绍列表（数组） |
| relationships | JSON | 关系列表 |
| rule_refs | JSON | 关联规则 ID 列表 |

### 9.2 索引设计

```sql
CREATE INDEX idx_events_project ON events(project_id);
CREATE INDEX idx_events_sort ON events(project_id, sort_order);
CREATE INDEX idx_entities_project ON entities(project_id);
```

### 9.3 存储策略

- **JSON 文件存储**：开发阶段以 JSON 文件为主，便于 Git 版本管理和人工审查
- **SQLite 存储**：单机使用时的结构化存储
- **MinIO 预备**：团队协作场景下的对象存储（可选）

## 十、缓存策略

### 10.1 缓存层级

| 缓存层级 | 内容 | TTL | 命中率目标 |
|----------|------|-----|-----------|
| **L1: 设定框架缓存** | worldview_rules, writing_rules, time_system | 长期（项目根配置不变即有效） | 高 |
| **L2: 事件上下文缓存** | 当前事件的完整 JSON + 相邻事件摘要 | 会话级 | 高 |
| **L3: DeepSeek 上下文缓存** | system prompt + 当前批次事件展开指令 | API 调用级 | >90% |

### 10.2 Redis Key 设计

```
sandbox:{project_id}:settings        — 项目设定缓存
sandbox:{project_id}:event:{ev_id}   — 单个事件缓存
sandbox:{project_id}:entity:{ch_id}  — 单个实体缓存
sandbox:{project_id}:spine           — 叙事链顺序缓存
sandbox:{project_id}:ir              — 中间表示缓存（编译管线产物）
```

### 10.3 缓存策略

- **设定框架缓存**：项目保存时更新，读取时优先使用缓存，数据库兜底
- **上下文对齐缓存**：DeepSeek 的 system prompt 前半部分（世界设定 + 写作规则）长期缓存，后半部分（当前事件上下文）按需刷新
- **缓存失效**：当项目保存或节点被编辑时，清除对应缓存线路
