## 一、产品概述

### 1.1 产品定位

**叙事沙盘**（StorySandbox）是一个专门为小说创作者设计的 AI 原生工具。

它不是为了"让 AI 替你写"，而是为了"**让你用 AI 把故事搭出来**"。

### 1.2 解决的核心问题

| 问题 | 叙事沙盘的解决方案 |
|------|---------------------|
| 通用 AI 生成内容逻辑混乱 | 用户通过拖拽事件节点**显式定义**叙事顺序，AI 只负责填充血肉 |
| 角色行为不符合人设 | 模拟沙盘中角色**基于设定自主推演**，不合理路径自动回退 |
| 大纲与成文脱节 | 大纲是**可执行的叙事蓝图**，AI 严格对齐生成 |
| API 调用成本高 | 小模型做结构性工作 + 缓存命中率 >90%，10 万字小说成本可压至几元 |

### 1.3 一句话口号

> **不是让 AI 替你写，而是让你用 AI 把故事搭出来。**

## 二、核心理念

### 2.1 四层架构

```
┌─────────────────────────────────────────────────────────────┐
│                    第一层：输入与梳理                         │
│   用户上传任何文本 → 小型模型提取 → 结构化设定/人物/世界观      │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    第二层：叙事链编辑                         │
│   事件节点可拖拽 → spine 数组定义叙事顺序 → 形成因果事件网络      │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    第三层：模拟推演                           │
│   角色扮演式推演 → 第三方观察者评估 → 用户审查 → 锁定大纲       │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    第四层：编译管线 + AI 生成                 │
│   JSON 节点 → 编译为结构化 MD → DeepSeek 逐事件展开为小说正文    │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 四大核心机制

#### 机制一：事件有向图 + 实体库

故事被建模为**事件有向图**和**实体库**两个正交维度。

- **事件（Event）** = 发生了什么，是故事的最小叙事单元
- **实体（Entity）** = 谁参与，是故事中的"演员"
- 参与关系（participants）记录实体在事件中的**状态快照**和**影响变化**，所有参与记录连起来就是角色的**弧光**

事件通过三种正交关系组织：

| 关系类型 | 含义 | 表达方式 | 回答的问题 |
|---------|------|---------|-----------|
| **因果** | 事件 A 导致了事件 B | `in_edges[]` / `out_edges[]` | "为什么发生" |
| **包含** | 事件 A 包含事件 B（整体/部分） | `sub_events[]` | "属于哪个部分" |
| **参与** | 实体参与了事件 | `participants[]` | "谁在场，什么状态" |

三种关系完全正交，互不干扰，共同构成故事的全息结构。

#### 机制二：spine 主链（叙事顺序控制）

这是叙事沙盘最核心的设计——将"事件真实发生的时间"和"读者看到的顺序"分离。

| 层级 | 名称 | 含义 | 记录方式 |
|------|------|------|----------|
| 后台 | **故事时间轴** | 事件真实发生的先后 | `time` + `in_edges/out_edges` |
| 前台 | **叙事链（spine）** | 读者看到的顺序，可自由重排 | `project.json` 中的 `spine[]` 数组 |

**支持的非线性叙事手法**：倒叙（闪回）、插叙、平行叙事、时间跳跃——全都在拖拽间完成。

#### 机制三：模拟沙盘

模拟沙盘让角色"活"起来。你在一个事件范围内放入角色、设定初始条件，角色基于设定自主推演行动，第三方观察者实时评估每一步的合理性。

| 组件 | 职责 | 实现方式 |
|------|------|----------|
| **角色 AI** | 让角色基于设定和情境自主行动 | 调用小模型进行角色扮演 |
| **第三方观察者** | 实时评估合理性，决定是否回退 | 硬约束/软约束分级判断 |
| **用户审查** | 最终决定是否采纳模拟结果 | 可视化界面，可编辑/重新模拟 |

```
角色A 输出行动 → 第三方观察者评估 → 合理？→ 更新场景状态 → 下一轮
                                  ↘ 不合理？→ 自动回退，尝试替代路径
                                                    ↘ 超过3次？→ 请求用户手动介入
```

#### 机制四：编译管线（从 JSON 到小说正文）

这是将结构化数据转化为小说正文的技术桥梁：

```
nodes/*.json + chars/*.json + project.json
                     ↓
       编译管线（加载 → 展开 → 验证 → IR → 生成）
                     ↓
        结构化 MD 字节码（一份完整的 Markdown 文件）
                     ↓
          DeepSeek（逐事件展开，填充血肉，生成正文）
```

- 大纲生成**不需要大模型**——用户可无限次调整，零成本
- 只有最终正文生成才调用 DeepSeek，成本极度可控
- 编译管线包含验证阶段，自动检测循环引用、时间矛盾、因果断裂

## 三、用户画像与场景

### 3.1 目标用户

| 类型 | 特征 | 使用场景 |
|------|------|----------|
| 网文作者 | 日更需求，常卡文 | 用沙盘推演后续情节，解决卡文 |
| 小说爱好者 | 有脑洞但不会写大纲 | 拖拽节点快速搭建故事框架 |
| 创意写作者 | 追求非线性叙事 | 自由调整叙事顺序，尝试不同结构 |
| 编剧/策划 | 需要多人协作 | 导出大纲用于团队讨论 |

### 3.2 典型使用流程

```
1. 创作者上传已有的设定、随笔、番外（或从零开始）
       ↓
2. 系统自动提取人物、世界观、规则，生成初始节点
       ↓
3. 创作者拖拽事件节点，编辑细节，定义叙事顺序（spine）
       ↓
4. 进入模拟沙盘，让角色"演"出关键情节，观察者实时反馈
       ↓
5. 用户审查推演结果，锁定大纲
       ↓
6. 编译管线将 JSON 节点编译为结构化 MD 大纲
       ↓
7. DeepSeek 基于 MD 大纲逐事件生成小说正文
       ↓
8. 不满意可回退到任意步骤重新调整
```

## 四、核心工作流

### 4.1 节点编辑器

叙事链编辑器是叙事沙盘的主工作区，一个可视化节点画布。

| 功能点 | 描述 |
|--------|------|
| 节点拖拽 | 自由调整 spine 数组顺序，所见即所得 |
| 三种事件类型 | event（普通）/ container（纯容器/卷/章）/ mixed（既有自身内容又有子事件） |
| 层次折叠 | 父事件可展开/折叠其子事件列表 |
| 锁定功能 | 锁定某些节点不允许 AI 改变（🔒图标） |
| 分支/合并 | 支持多线叙事，节点可汇聚/分叉 |

### 4.2 模拟沙盘

| 功能点 | 描述 |
|--------|------|
| 两种推演场景 | 填补空白（两事件间）/ 延伸剧情（父事件内部细化） |
| 角色扮演 | 轮流调用 AI 扮演每个角色，输出行动描述 |
| 第三方观察者 | 硬约束/软约束分级评估，自动回退不合理路径 |
| 手动介入 | 用户可在任意轮次打断，修改角色行动 |
| 三种导出方式 | 覆盖原事件 / 作为子事件 / 作为新事件 |

### 4.3 编译生成

| 功能点 | 描述 |
|--------|------|
| 零成本大纲 | 编译管线无需大模型，用户可无限次调整 |
| 严格对齐 | DeepSeek 基于大纲逐事件生成，不添加未授权事件 |
| 增量续写 | 支持在已有章节后续写 |
| 验证检查 | 自动检测循环引用、时间不一致、因果断裂 |

## 五、特色功能详解

### 5.1 三种事件类型

| type | 含义 | 示例 |
|------|------|------|
| **event** | 普通事件，没有子事件 | "遇刺"、"朝堂对峙" |
| **container** | 纯容器事件，自身不包含内容，只作为子事件的标题/分组 | "第一卷·风起"（只起标题作用） |
| **mixed** | 既有自己的内容，又包含子事件 | "酒馆冲突"（父事件有概括描述，子事件有具体细节） |

> container 类型统一替代了传统"故事段"的概念，无需独立类型。

### 5.2 叙事链 vs 时间轴

```
    包含树                   因果链
    ───────                  ───────
  第一卷·风起
  ├── 第一章·入京            ev_出发 → ev_遇刺 → ev_朝堂
  │   ├── ev_出发
  │   ├── ev_遇刺
  │   └── ev_朝堂
  └── 第二章·立威            ev_新政 → ev_受阻
      ├── ev_新政
      └── ev_受阻
```

- **竖着看**（包含树）：第一卷包含两章，每章包含几个事件——`sub_events[]` 表达
- **横着看**（因果链）：出发→遇刺→朝堂→新政→受阻——`in_edges/out_edges` 表达
- **前后看**（叙事顺序）：spine 数组决定读者看事件的顺序，可任意重排

### 5.3 模拟沙盘推演范围

| 场景 | 范围 | 输入 | 输出 |
|------|------|------|------|
| **填补空白** | 两个已有事件节点之间 | 前事件的 `outcome` + 后事件的 `in_edges` | 中间发生的新事件节点 |
| **延伸剧情** | 某个父事件内部 | 父事件的 `process` + `participants` | 拆解为 `sub_events` 列表 |

### 5.4 第三方观察者判断规则

```yaml
硬约束（不可违反）:
  - 主角不能死于非主线冲突（用户可预设）
  - 角色 A 和 角色 B 不能提前相遇（用户可预设）
  - 时间不能倒流（除非世界观设定允许）
  - 角色不能做出与自身核心目标完全相反的行为

软约束（可违反但需标记）:
  - 角色行为是否符合性格设定
  - 因果链是否成立
  - 情感转变是否有铺垫
  - 对话风格是否符合角色身份

自动回退触发条件:
  - 违反硬约束 → 立即回退
  - 连续 3 轮无明显进展 → 回退
  - 角色输出与自身设定目标矛盾 → 回退
  - 角色之间的行动逻辑冲突 → 回退
```

### 6.1 用户输入

用户只需要写一段世界观设定：

```
向日葵地：
暮色永驻之地。
大裂谷的一种生态，这里有很多机械结构的向日葵，像是精密的机械造物，
但是也像生物那样会生长和繁殖。
它们对着"太阳的一边"，整个生态都维持在暮色的阴影下。

后续研究发现向日葵中间的一小块玻璃是投影仪结果，
实际上在向日葵地看到的太阳实际上是许多向日葵向同一个方向投影出来的。

不要在那里单独行动！！！！
至今为此所有单独行动或者落单的人都失踪了，
只有部分向日葵会不断重播他们失踪前求救的录音，
```

## 七、技术亮点

### 7.1 模型分层调用，成本极低

| 阶段 | 模型 | 说明 |
|------|------|------|
| 结构化提取 | 小模型（7B） | 一次性处理，成本极低 |
| 模拟沙盘-角色扮演 | 小模型（7B） | 每轮每个角色，任务简单 |
| 模拟沙盘-观察者 | 小模型（7B） | 每轮，任务简单 |
| 编译管线（MD 生成） | **无需模型** | 本地递归遍历，零成本 |
| spine 因果验证 | **无需模型** | 本地算法验证，零成本 |
| 最终正文生成 | 大模型（DeepSeek） | 按章节生成，可控 |

**核心优势**：正文生成前不需要大模型，用户可以无限次调整大纲、推演剧情，直到满意才调用大模型生成正文。

### 7.2 缓存策略

- **设定框架缓存**：世界观、人物设定等稳定数据长期缓存
- **上下文对齐缓存**：DeepSeek 的 system prompt + 当前事件上下文缓存，高命中率
- **目标命中率**：>90%

### 7.3 验证体系

编译管线内置完整的验证检查：

| 检查项 | 说明 |
|--------|------|
| 引用完整性 | 检查 sub_events 指向的事件是否存在 |
| 时间一致性 | 检查因果链上的时间顺序是否矛盾 |
| 循环引用检测 | 检查 sub_events 是否形成循环 |
| 实体状态一致性 | 检查实体状态变化是否合理 |
| 因果链断裂检测 | 检测事件链中是否存在断开的因果引用 |

## 九、术语表

| 术语 | 含义 |
|------|------|
| **事件（Event）** | 故事的最小叙事单元，一个有向图节点 |
| **实体（Entity）** | 参与事件的"演员"（人物/物品/组织/概念） |
| **spine（主链）** | 记录叙事顺序的事件 ID 数组，控制读者看到事件的顺序 |
| **因果链** | 通过 `in_edges` / `out_edges` 表达的事件因果关系 |
| **sub_events** | 事件的子事件列表，表达整体/部分的包含关系 |
| **participants** | 事件的参与者列表，记录实体在事件中的状态和影响 |
| **container** | 一种事件类型，只作为容器/标题，本身不包含内容 |
| **mixed** | 一种事件类型，既有自身内容又包含子事件 |
| **第三方观察者** | 模拟沙盘中评估角色行为合理性的 AI 组件 |
| **编译管线** | 将 JSON 节点"编译"为结构化 MD 文档的处理流程 |
| **结构化 MD 字节码** | 编译管线生成的完整 MD 文件，可直接喂给 DeepSeek |
| **双层层叙事结构** | 将故事时间轴与叙事链顺序分离的设计模式 |
